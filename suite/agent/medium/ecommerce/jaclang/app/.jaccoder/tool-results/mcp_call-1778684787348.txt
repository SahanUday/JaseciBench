  "content": "# jac-client Reference\n\njac-client adds client-side compilation to Jac so you can write React-style UI components using `to cl:` section headers (or `.cl.jac` files). The compiler separates your code automatically -- server-side logic compiles to Python, while client-side components compile to JavaScript with React as the rendering engine.\n\nYou also get project scaffolding (`jac create --use client`), npm dependency management, a Vite-powered dev server with HMR, and automatic HTTP bridge generation so your client components can call server walkers without manual API wiring. This reference covers installation, project structure, the module system, component authoring, and build configuration.\n\n---\n\n## Installation\n\n```bash\npip install jac-client\n```\n\n---\n\n## Project Setup\n\n### Create New Project\n\n```bash\njac create myapp --use client\ncd myapp\n```\n\n### Project Structure\n\n```\nmyapp/\n\u251c\u2500\u2500 jac.toml           # Project configuration\n\u251c\u2500\u2500 main.jac           # Entry point with app() function\n\u251c\u2500\u2500 components/        # Reusable components\n\u2502   \u2514\u2500\u2500 Button.tsx     # TypeScript components supported\n\u2514\u2500\u2500 styles/            # CSS files\n    \u2514\u2500\u2500 main.css\n```\n\n### The `.cl.jac` Convention\n\nFiles ending in `.cl.jac` are automatically treated as client-side code -- no `to cl:` header needed:\n\n```jac\n# components/Header.cl.jac -- automatically client-side\ndef:pub Header() -> JsxElement {\n    return <header>My App</header>;\n}\n```\n\nThis is equivalent to starting a regular `.jac` file with a `to cl:` section header.\n\n---\n\n## Module System\n\nJac's module system bridges Python and JavaScript ecosystems. You can import from PyPI packages on the server and npm packages on the client using familiar syntax. The `include` statement (like C's `#include`) merges code directly, which is useful for splitting large files.\n\n### Import Statements\n\n```jac\n# Simple import\nimport math;\nimport sys, json;\n\n# Aliased import\nimport datetime as dt;\n\n# From import\nimport from typing { List, Dict, Optional }\nimport from math { sqrt, pi, log as logarithm }\n\n# Relative imports\nimport from . { sibling_module }\nimport from .. { parent_module }\nimport from .utils { helper_function }\n\n# npm package imports (client-side)\nimport from react { useState, useEffect }\nimport from \"@mui/material\" { Button, TextField }\n\n# CSS and asset imports\nimport \"./styles.css\";\nimport \"./global.css\";\n```\n\n### Include Statements\n\nInclude merges code directly (like C's `#include`):\n\n```jac\ninclude utils;  # Merges utils.jac into current scope\n```\n\n### Export and Visibility\n\n```jac\n# Public by default\ndef helper -> int { return 42; }\n\n# Explicitly public\ndef:pub api_function -> None { }\n\n# Private to module\ndef:priv internal_helper -> None { }\n\n# Public walker (becomes API endpoint with jac start)\nwalker:pub GetUsers { }\n\n# Private walker\nwalker:priv InternalProcess { }\n```\n\n---\n\n## Server-Side Development\n\n### Server Sections\n\n```jac\nto sv:\n\n# Server-only section\nnode User {\n    has email: str;\n}\n\n# Single-statement form (no header, no braces)\nsv import from .database { connect_db }\nsv node SecretData { has value: str; }\n```\n\n> **Note on `sv import` between two server modules.** When both the importer and the importee are server-context modules running as separate microservices, `sv import` generates HTTP client stubs instead of pulling the provider into the consumer's process. The same source also works as a monolith. See [Microservice Interop (sv-to-sv)](jac-scale.md#microservice-interop-sv-to-sv) in the jac-scale reference for details.\n\n### REST API with jac start\n\nPublic walkers automatically become REST endpoints:\n\n```jac\nwalker:pub GetUsers {\n    can get with Root entry {\n        users = [-->][?:User];\n        report users;\n    }\n}\n\n# Endpoint: POST /walker/GetUsers\n```\n\nStart the server:\n\n```bash\njac start main.jac --port 8000\n```\n\n### Typed Object Passing\n\nObjects crossing the server/client boundary are automatically serialized and hydrated as typed instances. You can return typed objects directly from server functions and walkers instead of manually constructing dicts:\n\n```jac\nnode Task {\n    has title: str,\n        done: bool = False;\n}\n\n# Server: return typed objects directly\ndef:pub get_tasks -> list[Task] {\n    return [root()-->][?:Task];\n}\n\ndef:pub create_task(title: str) -> Task {\n    task = root() ++> Task(title=title);\n    return task[0];\n}\n\n# Client: receives hydrated Task instances\nto cl:\n\nsv import from .main { get_tasks, create_task }\n\ndef:pub app -> JsxElement {\n    has tasks: list = [];\n\n    async can with entry {\n        tasks = await get_tasks();  # list of Task objects\n    }\n\n    async def addTask(title: str) -> None {\n        task = await create_task(title);  # a Task object\n        tasks = tasks + [task];\n    }\n\n    return <div>\n        {[<span key={t.title}>{t.title} - {t.done}</span> for t in tasks]}\n    </div>;\n}\n```\n\nThe compiler generates JavaScript class stubs with `__from_wire`/`__to_wire` methods for each type that crosses the boundary. This works for:\n\n- **`obj` types** -- fields are hydrated recursively (nested objects are also typed)\n- **`node` types** -- same as obj, plus graph identity is preserved (access via `jid(node)`)\n- **`enum` types** -- emitted as frozen JavaScript objects\n- **`list[T]` returns** -- each element is individually hydrated\n- **Bidirectional** -- typed objects sent as function arguments or walker `has` fields are serialized with `__type__` metadata and deserialized on the server\n\nWalker reports also benefit from typed hydration:\n\n```jac\nwalker:pub create_todo {\n    has text: str;\n\n    can create with Root entry {\n        new_todo = here ++> Task(title=self.text);\n        report new_todo;  # Client receives a typed Task, not a raw dict\n    }\n}\n```\n\n### Module Introspection\n\n```jac\nwith entry {\n    # List all walkers in module\n    walkers = get_module_walkers();\n\n    # List all functions\n    functions = get_module_functions();\n}\n```\n\n---\n\n## Client Sections\n\nUse the `to cl:` section header to tag every following module-level element as client-side (React) code:\n\n```jac\nto cl:\n\ndef:pub app() -> JsxElement {\n    return <div>\n        <h1>Hello, World!</h1>\n    </div>;\n}\n```\n\nA section header applies until the next `to X:` header or end of file. You can switch back with `to sv:`, `to na:`, or end the file.\n\n### Single-Statement Forms\n\nFor one-off client-side declarations, use the single-statement `cl` prefix:\n\n```jac\ncl import from react { useState }\ncl glob THEME: str = \"dark\";\n```\n\nThis also works for component definitions -- the preferred shorthand for a single tagged declaration inside a mostly-server file:\n\n```jac\ncl def:pub app -> JsxElement {\n    has count: int = 0;\n    return <div>Count: {count}</div>;\n}\n```\n\n### Braced Blocks (legacy / inner-scope)\n\nThe older `cl { ... }` braced block still works and is useful for **inner-scope overrides** inside a function or class, but at module scope it emits **W0064** pointing at the section-header form. In `.cl.jac` files or after a `to cl:` header, no wrapper is needed at all.\n\n### Export Requirement\n\nThe entry `app()` function must be exported with `:pub`:\n\n```jac\nto cl:\n\ndef:pub app() -> JsxElement {  # :pub required\n    return <App />;\n}\n```\n\n---\n\n## Components\n\n### Function Components\n\n```jac\nto cl:\n\ndef:pub Button(props: dict) -> JsxElement {\n    return <button\n        className={props.get(\"className\", \"\")}\n        onClick={props.get(\"onClick\")}\n    >\n        {props.children}\n    </button>;\n}\n```\n\n### Using Props\n\n```jac\nto cl:\n\ndef:pub Card(props: dict) -> JsxElement {\n    return <div className=\"card\">\n        <h2>{props[\"title\"]}</h2>\n        <p>{props[\"description\"]}</p>\n        {props.children}\n    </div>;\n}\n```\n\n### Composition\n\n```jac\nto cl:\n\ndef:pub app() -> JsxElement {\n    return <div>\n        <Card title=\"Welcome\" description=\"Hello!\">\n            <Button onClick={lambda -> None { print(\"clicked\"); }}>\n                Click Me\n            </Button>\n        </Card>\n    </div>;\n}\n```\n\n---\n\n## Reactive State\n\n### The `has` Keyword\n\nInside client-tagged code (`to cl:` sections, `.cl.jac` files, or `cl { }` blocks), `has` creates reactive state:\n\n```jac\nto cl:\n\ndef:pub Counter() -> JsxElement {\n    has count: int = 0;  # Compiles to useState(0)\n\n    return <div>\n        <p>Count: {count}</p>\n        <button onClick={lambda -> None { count = count + 1; }}>\n            Increment\n        </button>\n    </div>;\n}\n```\n\n### How It Works\n\n| Jac Syntax | React Equivalent |\n|------------|------------------|\n| `has count: int = 0` | `const [count, setCount] = useState(0)` |\n| `count = count + 1` | `setCount(count + 1)` |\n\n### Complex State\n\n```jac\nto cl:\n\ndef:pub Form() -> JsxElement {\n    has name: str = \"\";\n    has items: list = [];\n    has data: dict = {\"key\": \"value\"};\n\n    # Create new references for lists/objects\n    def add_item(item: str) -> None {\n        items = items + [item];  # Concatenate to new list\n    }\n\n    return <div>Form</div>;\n}\n```\n\n!!! warning \"Immutable Updates for Lists and Objects\"\n    State updates must produce new references to trigger re-renders. Mutating in place will not work.\n\n    ```jac\n    # Correct - creates new list\n    todos = todos + [new_item];\n    todos = [t for t in todos if t[\"id\"] != target_id];\n\n    # Wrong - mutates in place (no re-render)\n    todos.append(new_item);\n    ```\n\n---\n\n## React Hooks\n\n### useEffect (Automatic)\n\nSimilar to how `has` variables automatically generate `useState`, the `can with entry` and `can with exit` syntax automatically generates `useEffect` hooks:\n\n| Jac Syntax | React Equivalent |\n|------------|------------------|\n| `can with entry { ... }` | `useEffect(() => { ... }, [])` |\n| `async can with entry { ... }` | `useEffect(() => { (async () => { ... })(); }, [])` |\n| `can with exit { ... }` | `useEffect(() => { return () => { ... }; }, [])` |\n| `can with [dep] entry { ... }` | `useEffect(() => { ... }, [dep])` |\n| `can with (a, b) entry { ... }` | `useEffect(() => { ... }, [a, b])` |\n\n```jac\nto cl:\n\ndef:pub DataLoader() -> JsxElement {\n    has data: list = [];\n    has loading: bool = True;\n\n    # Run once on mount (async with IIFE wrapping)\n    async can with entry {\n        data = await fetch_data();\n        loading = False;\n    }\n\n    # Cleanup on unmount\n    can with exit {\n        cleanup_subscriptions();\n    }\n\n    return <div>...</div>;\n}\n\ndef:pub UserProfile(userId: str) -> JsxElement {\n    has user: dict = {};\n\n    # Re-run when userId changes (dependency array)\n    async can with [userId] entry {\n        user = await fetch_user(userId);\n    }\n\n    # Multiple dependencies using tuple syntax\n    async can with (userId, refresh) entry {\n        user = await fetch_user(userId);\n    }\n\n    return <div>{user.name}</div>;\n}\n```\n\n### useEffect (Manual)\n\nYou can also use `useEffect` manually by importing it from React:\n\n```jac\nto cl:\n\nimport from react { useEffect }\n\ndef:pub DataLoader() -> JsxElement {\n    has data: list = [];\n    has loading: bool = True;\n\n    # Run once on mount\n    useEffect(lambda -> None {\n        fetch_data();\n    }, []);\n\n    # Run when dependency changes\n    useEffect(lambda -> None {\n        refresh_data();\n    }, [some_dep]);\n\n    return <div>...</div>;\n}\n```\n\n### useContext\n\n```jac\nto cl:\n\nimport from react { createContext, useContext }\n\nglob AppContext = createContext(None);\n\ndef:pub AppProvider(props: dict) -> JsxElement {\n    has theme: str = \"light\";\n\n    return <AppContext.Provider value={{\"theme\": theme}}>\n        {props.children}\n    </AppContext.Provider>;\n}\n\ndef:pub ThemedComponent() -> JsxElement {\n    ctx = useContext(AppContext);\n    return <div className={ctx.theme}>Content</div>;\n}\n```\n\n### Custom Hooks\n\nCreate reusable state logic by defining functions that use `has`:\n\n```jac\nto cl:\n\nimport from react { useEffect }\n\ndef use_local_storage(key: str, initial_value: any) -> tuple {\n    has value: any = initial_value;\n\n    useEffect(lambda -> None {\n        stored = localStorage.getItem(key);\n        if stored {\n            value = JSON.parse(stored);\n        }\n    }, []);\n\n    useEffect(lambda -> None {\n        localStorage.setItem(key, JSON.stringify(value));\n    }, [value]);\n\n    return (value, lambda v: any -> None { value = v; });\n}\n\ndef:pub Settings() -> JsxElement {\n    (theme, set_theme) = use_local_storage(\"theme\", \"light\");\n    return <div>\n        <p>Current: {theme}</p>\n        <button onClick={lambda -> None { set_theme(\"dark\"); }}>Dark</button>\n    </div>;\n}\n```\n\n---\n\n## Backend Integration\n\n### Calling Walkers from Client\n\nUse native Jac `spawn` syntax to call walkers from client code. First, import your walkers with `sv import`, then spawn them:\n\n```jac\n# Import walkers from backend\nsv import from ...main { get_tasks, create_task }\n\nto cl:\n\ndef:pub TaskList() -> JsxElement {\n    has tasks: list = [];\n    has loading: bool = True;\n\n    # Fetch data on component mount\n    async can with entry {\n        result = root() spawn get_tasks();\n        if result.reports and result.reports.length > 0 {\n            tasks = result.reports[0];\n        }\n        loading = False;\n    }\n\n    if loading {\n        return <p>Loading...</p>;\n    }\n\n    return <ul>\n        {[<li key={task[\"id\"]}>{task[\"title\"]}</li> for task in tasks]}\n    </ul>;\n}\n```\n\n### Walker Response\n\nThe `spawn` call returns a result object:\n\n| Property | Type | Description |\n|----------|------|-------------|\n| `result.reports` | list | Data reported by walker via `report` |\n| `result.status` | int | HTTP status code |\n\n### Spawn Syntax\n\n| Syntax | Description |\n|--------|-------------|\n| `root() spawn WalkerName()` | Spawn walker from root node |\n| `root() spawn WalkerName(arg=value)` | Spawn with parameters |\n| `node_id spawn WalkerName()` | Spawn from specific node |\n\nThe spawn call returns a result object with:\n\n- `result.reports` - Data reported by the walker\n- `result.status` - HTTP status code\n\n### Mutations (Create, Update, Delete)\n\n```jac\nsv import from ...main { add_task, toggle_task, delete_task }\n\nto cl:\n\ndef:pub TaskManager() -> JsxElement {\n    has tasks: list = [];\n\n    # Create\n    async def handle_add(title: str) -> None {\n        result = root() spawn add_task(title=title);\n        if result.reports and result.reports.length > 0 {\n            tasks = tasks + [result.reports[0]];\n        }\n    }\n\n    # Update\n    async def handle_toggle(task_id: str) -> None {\n        result = root() spawn toggle_task(task_id=task_id);\n        if result.reports and result.reports[0][\"success\"] {\n            tasks = [\n                {**t, \"completed\": not t[\"completed\"]} if t[\"id\"] == task_id else t\n                for t in tasks\n            ];\n        }\n    }\n\n    # Delete\n    async def handle_delete(task_id: str) -> None {\n        result = root() spawn delete_task(task_id=task_id);\n        if result.reports and result.reports[0][\"success\"] {\n            tasks = [t for t in tasks if t[\"id\"] != task_id];\n        }\n    }\n\n    return <div>...</div>;\n}\n```\n\n### Error Handling Pattern\n\nWrap spawn calls in try/catch and track loading/error state:\n\n```jac\nto cl:\n\ndef:pub SafeDataView() -> JsxElement {\n    has data: any = None;\n    has loading: bool = True;\n    has error: str = \"\";\n\n    async can with entry {\n        loading = True;\n        try {\n            result = root() spawn get_data();\n            if result.reports and result.reports.length > 0 {\n                data = result.reports[0];\n            }\n        } except Exception as e {\n            error = f\"Failed to load: {e}\";\n        }\n        loading = False;\n    }\n\n    if loading { return <p>Loading...</p>; }\n    if error {\n        return <div>\n            <p>{error}</p>\n            <button onClick={lambda -> None { location.reload(); }}>Retry</button>\n        </div>;\n    }\n    return <div>{JSON.stringify(data)}</div>;\n}\n```\n\n### Polling for Real-Time Updates\n\nUse `setInterval` with effect cleanup for periodic data refresh:\n\n```jac\nto cl:\n\nimport from react { useEffect }\n\ndef:pub LiveData() -> JsxElement {\n    has data: any = None;\n\n    async def fetch_data() -> None {\n        result = root() spawn get_live_data();\n        if result.reports and result.reports.length > 0 {\n            data = result.reports[0];\n        }\n    }\n\n    async can with entry { await fetch_data(); }\n\n    useEffect(lambda -> None {\n        interval = setInterval(lambda -> None { fetch_data(); }, 5000);\n        return lambda -> None { clearInterval(interval); };\n    }, []);\n\n    return <div>{data and <p>Last updated: {data[\"timestamp\"]}</p>}</div>;\n}\n```\n\n---\n\n## Routing\n\n### File-Based Routing (Recommended)\n\njac-client supports file-based routing using a `pages/` directory:\n\n```\n\nmyapp/\n\u251c\u2500\u2500 main.jac\n\u2514\u2500\u2500 pages/\n    \u251c\u2500\u2500 index.jac          # /\n    \u251c\u2500\u2500 about.jac          # /about\n    \u251c\u2500\u2500 users/\n    \u2502   \u251c\u2500\u2500 index.jac      # /users\n    \u2502   \u2514\u2500\u2500 [id].jac       # /users/:id (dynamic route)\n    \u2514\u2500\u2500 (auth)/            # Route group (parentheses)\n        \u251c\u2500\u2500 layout.jac     # Shared layout for auth routes\n        \u251c\u2500\u2500 login.jac      # /login\n        \u2514\u2500\u2500 signup.jac     # /signup\n\n```\n\n**Route mapping:**\n\n| File | Route | Description |\n|------|-------|-------------|\n| `pages/index.jac` | `/` | Home page |\n| `pages/about.jac` | `/about` | Static page |\n| `pages/users/index.jac` | `/users` | Users list |\n| `pages/users/[id].jac` | `/users/:id` | Dynamic parameter |\n| `pages/[...notFound].jac` | `*` | Catch-all (404) |\n| `pages/(auth)/dashboard.jac` | `/dashboard` | Route group (no URL segment) |\n| `pages/layout.jac` | -- | Wraps child routes with `<Outlet />` |\n\nEach page file exports a `page` function:\n\n```jac\n# pages/users/[id].jac\ncl import from \"@jac/runtime\" { useParams, Link }\n\nto cl:\n\ndef:pub page() -> JsxElement {\n    params = useParams();\n    return <div>\n        <Link to=\"/users\">Back</Link>\n        <h1>User {params.id}</h1>\n    </div>;\n}\n```\n\n**Route groups** organize pages without affecting the URL. A layout file can wrap them with authentication:\n\n```jac\n# pages/(auth)/layout.jac -- protects all pages in this group\ncl import from \"@jac/runtime\" { AuthGuard, Outlet }\n\nto cl:\n\ndef:pub layout() -> JsxElement {\n    return <AuthGuard redirect=\"/login\">\n        <Outlet />\n    </AuthGuard>;\n}\n```\n\n### Manual Routes\n\nFor manual routing, import components from `@jac/runtime`:\n\n```jac\ncl import from \"@jac/runtime\" { Router, Routes, Route, Link }\n\nto cl:\n\ndef:pub app() -> JsxElement {\n    return <Router>\n        <nav>\n            <Link to=\"/\">Home</Link>\n            <Link to=\"/about\">About</Link>\n        </nav>\n\n        <Routes>\n            <Route path=\"/\" element={<Home />} />\n            <Route path=\"/about\" element={<About />} />\n        </Routes>\n    </Router>;\n}\n```\n\n### URL Parameters\n\n```jac\ncl import from \"@jac/runtime\" { useParams }\n\nto cl:\n\ndef:pub UserProfile() -> JsxElement {\n    params = useParams();\n    user_id = params[\"id\"];\n\n    return <div>User: {user_id}</div>;\n}\n\n# Route: /user/:id\n```\n\n### Programmatic Navigation\n\n```jac\ncl import from \"@jac/runtime\" { useNavigate }\n\nto cl:\n\ndef:pub LoginForm() -> JsxElement {\n    navigate = useNavigate();\n\n    async def handle_login() -> None {\n        success = await do_login();\n        if success {\n            navigate(\"/dashboard\");\n        }\n    }\n\n    return <button onClick={lambda -> None { handle_login(); }}>\n        Login\n    </button>;\n}\n```\n\n### Nested Routes with Outlet\n\n```jac\ncl import from \"@jac/runtime\" { Outlet }\n\nto cl:\n\n# pages/layout.jac -- root layout wrapping all pages\ndef:pub layout() -> JsxElement {\n    return <>\n        <nav>...</nav>\n        <main><Outlet /></main>\n        <footer>...</footer>\n    </>;\n}\n\n# pages/dashboard/layout.jac -- nested dashboard layout\ndef:pub DashboardLayout() -> JsxElement {\n    # Child routes render where Outlet is placed\n    return <div>\n        <Sidebar />\n        <main>\n            <Outlet />\n        </main>\n    </div>;\n}\n```\n\n### Routing Hooks Reference\n\nImport from `@jac/runtime`:\n\n| Hook | Returns | Usage |\n|------|---------|-------|\n| `useParams()` | dict | Access URL parameters: `params.id` |\n| `useNavigate()` | function | Navigate programmatically: `navigate(\"/path\")`, `navigate(-1)` |\n| `useLocation()` | object | Current location: `location.pathname`, `location.search` |\n| `Link` | component | Navigation: `<Link to=\"/path\">Text</Link>` built-in authentication functions via `@jac/runtime`.\n\n### Available Functions\n\n| Function | Returns | Description |\n|----------|---------|-------------|\n| `jacLogin(username, password)` | `bool` | Login user, returns True on success |\n| `jacSignup(username, password)` | `dict` | Register user, returns `{success: bool, error?: str}` |\n| `jacLogout()` | `void` | Clear auth token |\n| `jacIsLoggedIn()` | `bool` | Check if user is authenticated |\n\n**Additional user management operations** (available via API endpoints when using jac-scale):\n\n| Operation | Description |\n|-----------|-------------|\n| Update Username | Change username via API endpoint |\n| Update Password | Change password via API endpoint |\n| Guest Access | Anonymous user support via `__guest__` account |\n\n### jacLogin\n\n```jac\ncl import from \"@jac/runtime\" { jacLogin, useNavigate }\n\nto cl:\n\ndef:pub LoginForm() -> JsxElement {\n    has username: str = \"\";\n    has password: str = \"\";\n    has error: str = \"\";\n\n    navigate = useNavigate();\n\n    async def handleLogin(e: FormEvent) -> None {\n        e.preventDefault();\n        # jacLogin returns bool (True = success, False = failure)\n        success = await jacLogin(username, password);\n        if success {\n            navigate(\"/dashboard\");\n        } else {\n            error = \"Invalid credentials\";\n        }\n    }\n\n    return <form onSubmit={handleLogin}>...</form>;\n}\n```\n\n### jacSignup\n\n```jac\ncl import from \"@jac/runtime\" { jacSignup }\n\nto cl:\n\nasync def handleSignup() -> None {\n    # jacSignup returns dict with success key\n    result = await jacSignup(username, password);\n    if result[\"success\"] {\n        # User registered and logged in\n        navigate(\"/dashboard\");\n    } else {\n        error = result[\"error\"] or \"Signup failed\";\n    }\n}\n```\n\n### jacLogout / jacIsLoggedIn\n\n```jac\ncl import from \"@jac/runtime\" { jacLogout, jacIsLoggedIn }\n\nto cl:\n\ndef:pub NavBar() -> JsxElement {\n    isLoggedIn = jacIsLoggedIn();\n\n    def handleLogout() -> None {\n        jacLogout();\n        # Redirect to login\n    }\n\n    return <nav>\n        {isLoggedIn and (\n            <button onClick={lambda -> None { handleLogout(); }}>Logout</button>\n        ) or (\n            <a href=\"/login\">Login</a>\n        )}\n    </nav>;\n}\n```\n\n### Per-User Graph Isolation\n\nEach authenticated user gets an isolated root node:\n\n```jac\nwalker:pub GetMyData {\n    can get with Root entry {\n        # 'here' is the user-specific root node\n        my_data = [-->][?:MyData];\n        report my_data;\n    }\n}\n```\n\n### Single Sign-On (SSO)\n\nConfigure in `jac.toml`:\n\n```toml\n[plugins.scale.sso.google]\nclient_id = \"your-google-client-id\"\nclient_secret = \"your-google-client-secret\"\n```\n\n**SSO Endpoints:**\n\n| Endpoint | Description The full-stack Jac app -- frontend bundle, Jac runtime, and your backend walkers/functions -- ships as a single installer for Windows, macOS, and Linux. End users do not need Python or Node installed.\n\n**Architecture:**\n\nA desktop build produces a Tauri shell that hosts a webview pointed at a bundled **sidecar** -- a PyInstaller-frozen executable containing Python, jaclang, jac-client, your `.jac` sources, and any configured plugins. On launch, Tauri spawns the sidecar on a free local port, reads `JAC_SIDECAR_PORT=<port>` from its stdout, and injects the resulting URL into the webview before any page JavaScript runs. The webview is the same client bundle the web target produces; the sidecar is the same backend `jac start` would run, just frozen.\n\n**Prerequisites:**\n\n- Rust/Cargo: [rustup.rs](https://rustup.rs)\n- Platform build tools (Visual Studio Build Tools on Windows, Xcode Command Line Tools on macOS, `webkit2gtk` + `libssl` + `librsvg` on Linux)\n\n**Setup & Build:**\n\n```bash\n# 1. One-time setup (creates src-tauri/ directory)\njac setup desktop\n\n# 2. Development with hot reload\njac start main.jac --client desktop --dev\n\n# 3. Build installer for current platform\njac build --client desktop\n\n# 4. Build for specific platform\njac build --client desktop --platform windows\njac build --client desktop --platform macos\njac build --client desktop --platform linux\n```\n\n**Output:** Installers in `src-tauri/target/release/bundle/`:\n\n- Windows: `.exe` installer (NSIS) or `.msi`\n- macOS: `.dmg` or `.app` bundle\n- Linux: `.AppImage`, `.deb`, or `.rpm`\n\n**Configuration:** Window size, title, identifier, and other Tauri metadata are configured under `[desktop]` in `jac.toml` (the build regenerates `src-tauri/tauri.conf.json` from it on every build):\n\n```toml\n[desktop]\nname = \"MyApp\"\nidentifier = \"com.example.myapp\"\nversion = \"1.0.0\"\n\n[desktop.window]\ntitle = \"MyApp\"\nwidth = 1200\nheight = 800\nmin_width = 800\nmin_height = 600\n\n[desktop.platforms]\nwindows = true\nmacos = true\nlinux = true\n```\n\n**Python Dependencies:**\n\nDesktop builds automatically install and bundle Python dependencies from `jac.toml`:\n\n```toml\n[dependencies]\nwebsockets = \">=12.0\"\nhttpx = \">=0.27.0\"\n```\n\nThese are auto-installed into the bundling environment before PyInstaller runs -- no manual `pip install` needed. During the build, `JAC_BUILD=1` is set in the environment so any Jac code that auto-starts a server at import time stays inert (preventing port conflicts and unnecessary work).\n\n**Plugin Bundling:**\n\nDesktop builds bundle Jac plugins into the sidecar executable using PyInstaller's `collect_all()` plus `importlib.metadata.requires()` for transitive dependency discovery. Configure which plugins to include via `[desktop.plugins]` in `jac.toml`:\n\n```toml\n[desktop.plugins]\njac_scale = true   # jac-scale: FastAPI server, auth, persistence (default: true)\nbyllm = true       # byllm/litellm for LLM support (default: true)\njac_coder = true   # jac-coder: AI coding features (default: true)\njac_mcp = true     # jac-mcp: MCP server integration (default: true)\n```\n\n**Notes:**\n\n- Plugins must be installed (`pip install jac-scale byllm jac-coder jac-mcp`) before building -- the build collects them from your current Python environment.\n- `jac_client` itself is **always** bundled as a core package regardless of this config, because the sidecar entry point imports it directly. Setting `jac_client = false` is ignored.\n- The build excludes build-artifact directories (`src-tauri/`, `node_modules/`, `dist/`, `.jac/client/`) when collecting `.jac` files, so rebuilds do not recursively nest previous sidecar bundles.\n\n**Bundled Jac Sources:**\n\nAll `.jac` files, `jac.toml`, and the `assets/` directory are copied into `src-tauri/jac/` and shipped as Tauri bundle resources. At runtime, the sidecar looks up `main.jac` in this bundled location first, falling back to parent directories. This is what makes desktop installs fully self-contained.\n\n**Extra Data Files:**\n\nShip additional files into the sidecar bundle via `[desktop.bundle] extra_data` in `jac.toml`. Values are [glob patterns](https://docs.python.org/3/library/pathlib.html#pathlib.Path.glob) rooted at the project directory. Matches keep their relative paths inside the bundle, so `Path(__file__).parent / \"config/prompts.yaml\"` still resolves at runtime.\n\n```toml\n[desktop.bundle]\nextra_data = [\n    \"config/*.yaml\",\n    \"data/seed.json\",\n    \"prompts/**/*.txt\",\n]\n```\n\n#### Data Persistence on Installed Builds\n\nInstalled desktop apps live in **read-only** locations -- `/usr/lib/`, `/opt/`, `C:\\Program Files\\`, or an AppImage's `/tmp/.mount_AppXXX/` squashfs. The Jac runtime and jac-scale write to disk relative to the working directory by default (the SQLite database `database.db`, the `.jac/data/` directory, session files), and those writes will fail or crash on a read-only mount.\n\nThe sidecar resolves this at startup, **before** importing any Jac module, by setting the `JAC_DATA_PATH` environment variable to a writable location and `chdir`-ing into it. The Jac runtime's `get_db_path()` and jac-scale's config loader both honor this variable.\n\n**Default fallback chain** (the sidecar picks the first one that exists or can be created and passes a touch/unlink probe):\n\n| Platform | Default | First fallback | Second fallback |\n|----------|---------|----------------|-----------------|\n| Linux / macOS | `~/.local/share/jac-app` | `~/.jac-app` | `/tmp/jac-app-{uid}` |\n| Windows | `%LOCALAPPDATA%\\jac-app` | `~/AppData/Local/jac-app` | `%TEMP%\\jac-app` |\n\n**Override the default** by passing `--data-path` to the sidecar (useful when running the bundled sidecar binary by hand for debugging, or when wiring it into a launcher you control):\n\n```bash\n./src-tauri/binaries/jac-sidecar --data-path /var/lib/myapp\n```\n\nYou can also export `JAC_DATA_PATH` before launching the app to point at a custom location for that run. The path you choose must be writable by the user running the app -- the sidecar will probe it and fail loudly if it cannot.\n\n**AppImage-specific environment cleanup:** AppImage runtimes inject `PYTHONHOME`, `PYTHONPATH`, and `PYTHONDONTWRITEBYTECODE` into the environment, which break the bundled Python interpreter inside the sidecar. The generated Tauri `main.rs` strips these variables before spawning the sidecar process.\n\n#### Client-Only Mode\n\nFor setups where the desktop app is just a thin native shell around a remote backend (e.g., a hosted jac-scale deployment), set `client_only = true` under `[desktop]` in `jac.toml`:\n\n```toml\n[desktop]\nclient_only = true\n\n[plugins.client.api]\nbase_url = \"https://api.example.com\"\n```\n\nIn this mode the build:\n\n- **Skips sidecar bundling entirely** -- no PyInstaller step, no Python bundle, smaller installer.\n- **Requires** `[plugins.client.api] base_url` to be set; the build raises a `RuntimeError` if it is missing, since the webview has no local backend to talk to.\n- **Still produces a full Tauri installer** -- only the backend half is omitted.\n\nIt is also useful in CI, where you may want to verify the web bundle compiles inside a desktop build without paying for the PyInstaller round-trip.\n\n#### Runtime API URL Injection (Debugging)\n\nDesktop builds do **not** embed the API base URL at compile time. Tauri allocates the sidecar port dynamically, then injects `window.__JAC_API_BASE_URL__` into the webview via an `initialization_script` before any page JavaScript executes. A `get_api_url` Tauri command is also exposed as a fallback for code that needs to query the URL after page load.\n\nIf you are debugging an \"API not reachable\" issue inside an installed desktop app:\n\n1. This uses compiler-provided `endpoint_effects` metadata -- no manual cache annotations or `jacInvalidate()` calls needed.\n\n**How it works:**\n\n1. The compiler classifies each walker/function endpoint as a **reader** (no side effects) or **writer** (modifies state)\n2. Reader responses are stored in an LRU cache (500 entries, 60-second TTL)\n3. Concurrent identical requests are deduplicated (only one network call)\n4. When a writer endpoint is called, all cached reader responses are automatically invalidated\n5. Auth state changes (login/logout) clear the entire cache\n\nThis means spawning the same walker twice in quick succession only makes one API call, and creating/updating data automatically refreshes any cached reads.\n\n---\n\n## BrowserRouter (Clean URLs)\n\njac-client uses `BrowserRouter` for client-side routing, producing clean URLs like `/about` and `/users/123` instead of hash-based URLs like `#/about`.\n\nFor this to work in production, your server must return the SPA HTML for all non-API routes. When using `jac start`, this is handled automatically -- the server's catch-all route serves the SPA HTML for extensionless paths, excluding API prefixes (`cl/`, `walker/`, `function/`, `user/`, `static/`).\n\nThe Vite dev server is configured with `appType: 'spa'` for history API fallback during development.\n\n---\n\n## Build Error Diagnostics\n\nWhen client builds fail, jac-client displays structured error diagnostics instead of raw Vite/Rollup output. JacClientErrorBoundary\n\n`JacClientErrorBoundary` is a specialized error boundary component that catches rendering errors in your component tree, logs them, and displays a fallback UI, preventing the entire app from crashing when a descendant component fails.\n\n### Quick Start\n\nImport and wrap `JacClientErrorBoundary` around any subtree where you want to catch render-time errors:\n\n```jac\ncl import from \"@jac/runtime\" { JacClientErrorBoundary }\n\nto cl:\n\ndef:pub app() -> JsxElement {\n    return <JacClientErrorBoundary fallback={<div>Oops! Something went wrong.</div>}>\n        <MainAppComponents />\n    </JacClientErrorBoundary>;\n}\n```\n\n### Built-in Wrapping\n\nBy default, jac-client internally wraps your entire application with `JacClientErrorBoundary`. This means:\n\n- You don't need to manually wrap your root app component\n- Errors in any component are caught and handled gracefully\n- The app continues to run and displays a fallback UI instead of crashing\n\n### Props\n\n| Prop               | Type              | Description                          |\n|--------------------|-------------------|--------------------------------------|\n| `fallback`         | JsxElement        | Custom fallback UI to show on error  |\n| `FallbackComponent`| Component         | Show default fallback UI with error  |\n| `children`         | JsxElement        | Components to protect                |\n\n### Example with Custom Fallback\n\n```jac\nto cl:\n\ndef:pub App() -> JsxElement {\n    return <JacClientErrorBoundary fallback={<div className=\"error\">Component failed to load</div>}>\n        <ExpensiveWidget />\n    </JacClientErrorBoundary>;\n}\n```\n\n### Nested Boundaries\n\nYou can nest multiple error boundaries for fine-grained error isolation:\n\n```jac\nto cl:\n\ndef:pub App() -> JsxElement {\n    return <JacClientErrorBoundary fallback={<div>App error</div>}>\n        <Header />\n        <JacClientErrorBoundary fallback={<div>Content error</div>}>\n            <MainContent />\n        </JacClientErrorBoundary>\n        <Footer />\n    </JacClientErrorBoundary>;\n}\n```\n\nIf `MainContent` throws an error, only that boundary's fallback is shown, while `Header` and `Footer` continue rendering normally.\n\n### Use Cases\n\n1. **Isolate Failure-Prone Widgets**: Protect sections that fetch data, embed third-party code, or are unstable\n2. **Per-Page Protection**: Wrap top-level pages/routes to prevent one error from failing the whole app\n3. **Micro-Frontend Boundaries**: Nest boundaries around embeddables for fault isolation\n\n---\n\n## Memory & Persistence\n\n### Memory Hierarchy\n\n| Tier | Type | Implementation |\n|------|------|----------------|\n|
